## 1. High

### H2. Sync: remote deletions never propagate; deleted entities resurrect [verified]
`src/services/backup/sync.ts:255-260` (removed changes skipped — no filename), `:286-290` (`allRemoved` → plain `return`), `:907-951` (`queueLocalOnlyEntities`)

Drive `removed: true` changes carry no file metadata, so they can't be mapped to an entity and are dropped; even when mapped, the all-removed branch just returns. No device ever deletes local data in response to a remote delete. Worse, a later full reconcile on the other device sees the still-present entity as "local only" and **re-uploads it** — device A then re-downloads the clip it deleted.

**Failure:** delete a clip on device A → device B keeps it forever and eventually resurrects it remotely.

### H4. Sync: update-in-place has no 404 fallback — entity becomes permanently unsyncable once its remote file is gone [verified]
`src/services/backup/sync.ts:614,659,676,719`; `src/services/backup/drive.ts:299-302`

When the manifest has a `remote_file_id`, upload always uses `drive.updateFile()`, which throws on 404. Nothing invalidates the manifest entry or falls back to create-new. Reachable from inside the app via H2: device A deletes a clip (remote files permanently deleted), device B never learns of it, B's next edit/transcription of that clip 404s three times and is abandoned (see M7) — forever, since the manifest still points at the dead file id.

### H5. Books × sync: a sync-applied deletion leaves `hidden=1` with a live `uri` — audio leaks and re-adding "restores" nothing [verified]
`src/services/storage/database.ts:603-614` (`restoreBookFromBackup` `ON CONFLICT` updates `hidden` but never `uri`), `src/actions/load_file.ts:55` (branches on `existingBook?.uri` only), `load_file.ts:83-95` (`handleDuplicate` → `touchBook`)

`hideBook` locally always nulls `uri` and deletes the file; the sync path doesn't. After pulling a remote deletion the book has `hidden=1` **and** a live `uri` + file on disk — a state the codebase assumes impossible (BOOKS.md's three-state model has no such combination, and `getAllFileUris` includes the uri so orphan cleanup will never reclaim the file).

**Failure:** (a) deleting a book on device B leaks the full audio file on device A forever; (b) the user re-adds the same file on A to get the book back → fingerprint matches, `existingBook.uri` is non-null → **duplicate** path: `touchBook` bumps `updated_at` without clearing `hidden`, so the book stays invisible *and* the queued upsert re-propagates `hidden=1` with a newer timestamp to all devices — the restore documented in BOOKS.md ("re-adding the same file triggers a restore") actively cements the deletion instead.

## 2. Medium

### M1. Sessions are never finalized on book switch or clip-ownership change [verified — found independently by two reviewers]
`src/store/index.ts:197-217`

The early returns (`!playback.uri`, `ownerId !== MAIN_PLAYER_OWNER_ID`) sit *above* the `finalizeSession` branch, and `loadBook` nulls `playback.uri` during transitions, so finalization only runs on a plain pause while main still owns playback. Sub-1-second accidental sessions from a quick book switch are never pruned (the `MIN_SESSION_DURATION_MS` cleanup never runs for them), `currentSessionBookId` goes stale, and `ended_at` lags up to 5s.

### M2. Transcription: bounds edit during an in-flight transcription keeps the *old* audio's text — permanently, and it syncs [verified]
`src/services/transcription/queue.ts:171-177`, `src/actions/update_clip.ts:63,85-87`, `src/store/index.ts:247-257`

An in-flight job's result lands after `updateClip` cleared `transcription` and re-queued; the re-queued job is then skipped because the queue only processes clips whose `transcription IS NULL`. The stale text is written via the store's finish handler and queued for sync. *(The M3 fix removed the queue's direct DB write, but the finish-handler path still applies the stale result — the race itself is unfixed.)*

### M5. Sync: audio-only clip changes are dropped — other devices keep mismatched clip audio forever [reported]
`src/services/backup/sync.ts:349-355` (`if (!jsonChange?.file) return`), upload ordering `:657-677` (JSON before audio)

Device B syncing in the window between A's JSON and audio uploads applies new bounds with old audio; when the audio-only change later arrives, the group has no JSON change and is dropped; `isSameVersion` then skips the entity on every subsequent sync.

### M6. Migrations are non-atomic, and migration 0 records completion *before* creating the schema [reported]
`src/services/storage/database.ts:128-137, 807-831`

Migration 0's first statements create `status` and insert `migration = 0` before `files`/`clips`/`sessions` exist; nothing is transactional. App killed mid-migration-0 → next launch starts at migration 1 and the core tables are never created. App killed mid-migration-4 → re-run throws "duplicate column" in the constructor and the app can never start again.

### M7. Sync: outbox items that fail 3 times vanish from `pendingCount` — Settings shows "Up to date" while changes are permanently unsynced [verified]
`src/services/storage/database.ts:897-903` (`attempts < 3` filter), `src/services/backup/sync.ts:546`, `src/screens/SettingsScreen.tsx:88`

No surfacing and no recovery path (e.g. a clip whose audio exceeds the 50MB cap fails deterministically). The only signal is a transient `sync.error` cleared by the next successful sync.

### M8. Sync: clips of a fingerprint-deduped book are downloaded but invisible, and accumulate [reported]
`src/services/backup/sync.ts:448-452`, `src/services/storage/database.ts:511` (`INNER JOIN files`)

When a remote book's fingerprint matches a local book with a different id, the book is skipped but its clips are still downloaded with the *remote* `source_id`, which joins to nothing. Same audiobook imported independently on two devices → clips made on A never appear on B, but their rows and audio files pile up.

### M9. Sync: bootstrap failure still runs the push phase with an empty manifest — duplicate remote files [reported]
`src/services/backup/sync.ts:198-207` (init errors caught, pull returns) then `:172` (`pushOutbox` runs anyway); upload→manifest write also not atomic (`:615-623`)

Every queued entity is uploaded create-new even though it already exists remotely. Once duplicated, reconcile picks an arbitrary `.json` and update-in-place only ever refreshes one twin.

### M10. `cleanupOrphanedFiles` TOCTOU: can delete files whose DB rows don't exist yet [reported — flagged independently by three reviewers]
`src/actions/cleanup_orphaned_files.ts:17-26` (runs at the start of every `loadFile`), race partners: `src/services/backup/sync.ts:492-497` (clip audio written *before* `restoreClipFromBackup`), `src/actions/add_clip.ts:42-58` (slice before `createClip`)

Background sync downloading a clip while the user taps "Add file" → the fresh `.m4a` is deleted between write and insert; the manifest says the download succeeded, so it is never re-downloaded — a silently broken clip.

### M11. Cancel arriving after the copy finished silently adds the book anyway [verified]
`src/actions/cancel_load_file.ts`, `android/.../FileCopierModule.kt:246-253`, `src/actions/load_file.ts:175-183`

Native `cancelCopy` resolves `null` for an already-completed op; `loadFile` proceeds through `db.upsertBook` + `queueChange`. The opId-guarded updater suppresses only UI updates, not DB writes: the dialog closes as "cancelled", then the book appears in the library.

### M13. Archive/delete rollback is incomplete when `queueChange` fails — store and DB diverge [reported]
`src/actions/archive_book.ts:40-54`, `src/actions/delete_book.ts:40-54`

The try block spans both the DB mutation and `queueChange`; the catch rolls back only the store. The book looks active but the DB row has `uri = NULL`; the next `fetchBooks` makes it vanish, the file is orphaned, and the mutation never syncs.

### M14. Seeking while paused is never persisted [reported]
`src/store/index.ts:204` (only `updateBookPosition` call site), `src/actions/seek.ts`, `src/services/audio/player.ts:185,200`

Position persists only from playback status/progress events; `TrackPlayer.seekTo()` while paused emits neither. Pause → scrub 20 minutes → app killed → resume at the stale position (which also syncs to other devices).

### M16. Timeline: `_isDragging` stuck `true` after a pan+pinch gesture — timeline stops following playback [reported]
`src/components/timeline/engine.ts:379-381` (`panEnd` early-returns during pinch cooldown *before* clearing `_isDragging`), `:194-199`, `:218`, `:512-515`

ClipEditor only (`canZoom`): pan with one finger, add a second to pinch, lift both → `panEnd` lands inside the 200ms cooldown and never clears the flag; `setExternalPosition` permanently no-ops and tap-to-seek animations cancel on their first tick until a fresh pan resets it. Related: `panUpdate` doesn't check `_isDragging`, so a pan started during pinch resumes against a stale `_dragStartOffset` and jumps the scroll offset.

### M17. `slicer.move()` deletes the clip's only audio before the move — a failed move loses it permanently [reported]
`src/services/audio/slicer.ts:90-99`, `src/actions/update_clip.ts:52-67`

The destination is unlinked before `RNFS.moveFile`; if the move throws (disk full, I/O), the DB still references the now-nonexistent file and the temp slice is later reclaimed by orphan cleanup. If the source book is later archived, the clip silently loses the independence guarantee it exists to provide.

### M18. Duplicate detection is defeated when the file size is unknown (`-1`) [reported — flagged independently by two reviewers]
`android/.../FileCopierModule.kt:81` (`queryFileSize(uri) ?: -1L`), `:270-281` (`queryFileSize` only handles `content://` — returns null for `file://`), `src/actions/load_file.ts:52-53`

`getBookByFingerprint(-1, fp)` uses `WHERE file_size = ?` and can never match the stored real size. Re-adding an existing audiobook via a provider that doesn't report `OpenableColumns.SIZE` (some cloud providers/mail attachments) creates a full second copy, and the `-1` is synced and poisons dedup on other devices. The `file://` gap is latent today (the picker always yields `content://`) but breaks the module's own documented contract and any future share-intent entry point.

## 3. Low

- **L1. `queuePositionSync` throttle is global and leading-edge-only** — `src/store/index.ts:59-61`, `src/utils/index.ts:52-61`. Not keyed by book id, no trailing call: after a book switch the new book's first queue call is suppressed for up to 30s, and the final position before a pause is never queued at all (pause doesn't queue). Locally the DB is correct; other devices just never receive the last ≤30s / the switched-to book's position until a future mutation. *[verified — demonstrated with a node repro of `throttle`; flagged independently by two reviewers]*
- **L2. "Last played" is actually "last modified"** — `src/services/storage/database.ts:358-363` (`ORDER BY updated_at DESC`). Startup auto-load can load a book the user merely edited — or one that changed *on another device* via sync. *[reported]*
- **L3. Session track/finalize are un-awaited and can interleave** — `src/store/index.ts:213-216`. A pause landing between `trackSession`'s read and write lets a short session survive; the reverse order can queue an `upsert` after a `delete` for the same session (outbox UNIQUE semantics), resurrecting it remotely. *[reported]*
- **L7. Trashed Drive files are treated as live changes; full reconcile then duplicates them** — `src/services/backup/drive.ts:237` (changes feed doesn't filter `trashed`) vs `drive.ts:55` (`listFiles` does). *[reported]*
- **L8. Clip audio download builds base64 char-by-char on the JS thread** — `src/services/backup/sync.ts:973-988`; a large clip is a ~67MB string built one character at a time. The 50MB cap guards uploads only. Freeze/OOM risk. *[reported]*
- **L9. Concurrent first syncs can create duplicate `Ivy/` folder trees** — `src/services/backup/drive.ts:397-442` (search-then-create, `files[0]` thereafter). *[reported]*
- **L11. Cancel between `beginCopy` and `commitCopy` misclassified as a real error** — `FileCopierModule.kt` rejects `UNKNOWN_OP`, `isCancellation()` checks only `code === 'CANCELLED'` (`src/actions/load_file.ts:207-209`). Contained today (opId-guarded UI), but a trap. *[reported]*
- **L12. `src/utils/hooks.ts` (uncommitted) is dead code** — zero imports; LibraryScreen and ClipsListScreen still hand-roll the identical search/menu state it extracts. Wire it in or drop it before committing. *[verified]*
- **L13. Dead code with a latent bug** — `FileStorageService.copyToAppStorage`/`rename` have no callers; `createUniqueFilename` (`src/services/storage/files.ts:146-155`) mangles extension-less names. Harmless only because dead. *[reported]*

## 5. The yt-dlp removal branch itself: verified clean

Every hunted class of removal residue came up empty:

- **No dangling references** to the downloader/`load_from_url`/yt-dlp anywhere in `src/`, `app/`, `android/`, `maestro/` (only an explanatory test comment). `tsc` clean, tests green.
- **The `YoutubeDLException` shim is exactly right.** Two reviewers independently downloaded the published `ffmpeg-0.18.1.aar` from Maven Central and disassembled `FFmpeg.class`: the only references into `com.yausername.youtubedl_android` are the class `YoutubeDLException` and one methodref, `<init>(String, Throwable)` — which the shim provides (public, correct package). The only other external classes `FFmpeg` touches (`SharedPrefsHelper`, `ZipUtils`, commons-io `FileUtils`) are all satisfied by the POM's runtime deps (`common-0.18.1`, `commons-io`). `:library` is not in the POM, so the shim is both necessary and sufficient. - **`cancel_load_file`** correctly resets all four library fields and cancels only the copier; the opId-guarded updater prevents the background error path from clobbering the reset.
- **UI**: LibraryScreen menu/empty states and SettingsScreen are coherent post-removal; LibraryLoadingDialog degrades exactly as the plan predicted.
- **Packaging**: `expo.useLegacyPackaging=true` retained; `MainApplication.kt` registers the four remaining packages and not `FileDownloaderPackage`.
- **APK inspection** (see §6): `libffmpeg.so` present, shim class present in DEX, zero yt-dlp engine classes (`YoutubeDLRequest`, Python payloads) in DEX.

One nit: `AudioSlicerModule.kt`/`ChapterReaderModule.kt` still put the removed `packages/python|aria2c/usr/lib` dirs on `LD_LIBRARY_PATH` — harmless (missing entries are ignored), cosmetic cleanup only.

## 7. Open questions / qualifications

1. **L1 (position sync throttle)** may be a deliberate staleness tradeoff for single-book use; the cross-book suppression, however, looks unintended. Worth a decision rather than a silent fix.
2. **M14 (seek-while-paused not persisted)**: persisting on every seek would be trivial, but if the current behavior is intentional (avoid churning the outbox while scrubbing), a persist-on-app-background hook would cover the data-loss window instead.
3. The remaining sync findings (H2, H4, H5, M5, M7–M9) share one root: the push/pull phases assume more atomicity than Drive and the outbox provide. A design pass (deletion tombstones, manifest invalidation on 404, given-up-item surfacing, bootstrap gating) would address them as a family rather than as six patches.
4. The sync unit tests mock the DB entirely, which is how the outbox UNIQUE-key bug (fixed H1) and M7's visibility filter stayed invisible. A real-sqlite (or in-memory fake with real constraints) test harness for the outbox is recommended as part of any sync work.
5. Findings tagged *[reported]* were produced by a subsystem reviewer with file:line evidence and spot-checked for plausibility, but not line-by-line re-verified the way the High findings were. Treat the cited line numbers as accurate as of this commit.
